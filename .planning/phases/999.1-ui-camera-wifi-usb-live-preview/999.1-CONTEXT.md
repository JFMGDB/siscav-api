<domain>
## Phase Boundary

Entregar, **num cliente de interface** (web ou app de operador), a capacidade de:

1. **Ligar uma câmara por USB** ao posto do operador (navegador ou shell nativo) e **ver pré-visualização em tempo real**.
2. **Ligar uma câmara por Wi‑Fi** (tipicamente câmara IP com URL de stream ou app compatível) e **ver pré-visualização em tempo real**.

**Fora de âmbito nesta fase** (não são novas capacidades a planear aqui — backlog futuro):

- OCR / leitura automática de matrícula na UI (continua a poder ser feita noutro cliente ou fase).
- **Upload** do frame para `POST /api/v1/access_logs/` (a API já suporta; integração “capturar → enviar” é fase ou tarefa separada se não for explicitamente incluída ao planear).
- Gravação contínua, DVR, multi-câmara em mosaico complexo, WebRTC/TURN em produção (podem ser fases próprias).

A API **SISCAV neste repositório** permanece **HTTP/JSON/multipart**; streaming de vídeo **não** é requisito obrigatório do backend para o MVP de pré-visualização (tráfego pode ser direto browser↔câmara ou stream URL).

</domain>

<decisions>
## Implementation Decisions

### Onde corre a UI

- **D-01:** Frontend num **repositório Git separado** deste (`siscav-api`), **sem** `apps/web` neste monorepo como alvo principal.
- **D-01b:** Stack fixa: **Next.js** + **TypeScript**. Integração com a API SISCAV apenas via **HTTP** (JSON/multipart onde aplicável), **CORS** e variáveis de ambiente (URL base da API, etc.) — ver `docs/api/FRONTEND_INTEGRATION.md`.

### USB (browser)

- **D-02:** Usar **MediaDevices / `getUserMedia`** com **seletor de dispositivo** (`deviceId`) quando existir mais do que uma câmara.
- **D-03:** Pré-visualização num elemento **`<video>`** (autoplay muted, playsInline) — padrão web para stream local.
- **D-04:** **HTTPS** em produção (ou `localhost` em dev) — requisito dos browsers para getUserMedia; documentar para operadores.

### Wi‑Fi (câmara IP / rede)

- **D-05:** Tratar “Wi‑Fi” como **stream exposto por URL** configurável pelo utilizador (ex.: **MJPEG** `http(s)://.../mjpeg`, snapshot repetido, ou **HLS** se o browser suportar via `<video>` + tipo MIME adequado). A câmara e a rede devem ser **acessíveis ao cliente** que corre a UI (mesma LAN ou VPN).
- **D-06:** **Sem proxy de vídeo obrigatório na API FastAPI** no MVP — evita latência, estado no servidor e CORS complexo. Se no futuro o stream só for acessível sem CORS ou em rede fechada, prever **fase próxima**: proxy read-only ou túnel (não bloqueia o desenho desta fase).

### UX de ligação

- **D-07:** Dois caminhos explícitos na UI: **“USB / este dispositivo”** (permite browser pedir consentimento) e **“Câmara na rede (URL)”** (campo de URL + botão “Testar” / pré-visualização).
- **D-08:** Estados claros: sem permissão, a carregar stream, erro de rede/CORS, stream ativo.

### Integração com SISCAV (contexto apenas)

- **D-09:** Autenticação de operadores contra a API segue **`docs/api/FRONTEND_INTEGRATION.md`** (JWT access + refresh). A pré-visualização **não** exige token por si só; chamadas à API continuam com `Authorization: Bearer`.

### Claude's Discretion

- **App Router vs Pages Router**, estrutura de pastas, e design system / componentes UI dentro de Next.js.
- Bibliotecas auxiliares para **HLS** ou MJPEG em `<img>`/`<video>` se necessário.
- Textos, ícones, layout exato e tratamento fino de erros.

</decisions>
