## objective

Add **SonarCloud/SonarQube Server** analysis to GitHub Actions: generate **Cobertura XML** from pytest, add **`sonar-project.properties`**, and run **`SonarSource/sonarqube-scan-action@v5`** after tests pass. **Do not** enable `sonar.qualitygate.wait=true` in this plan (baseline onboarding — SONAR-03 policy documented in plan 02).

## must_haves

1. **`pytest` in CI** writes **`coverage.xml`** at the repository root (in addition to existing terminal coverage).
2. **`sonar-project.properties`** exists at repo root with `sonar.sources=apps`, `sonar.tests=tests`, `sonar.python.version=3.13`, `sonar.python.coverage.reportPaths=coverage.xml`, and placeholders **`sonar.organization`** + **`sonar.projectKey`** documented inline for SonarCloud onboarding.
3. **`.github/workflows/ci.yml`** includes a Sonar scan step using `SonarSource/sonarqube-scan-action@v5` with `SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}` and `SONAR_HOST_URL: ${{ vars.SONAR_HOST_URL }}` defaulting to `https://sonarcloud.io` when the variable is unset (use expression: `${{ vars.SONAR_HOST_URL || 'https://sonarcloud.io' }}` or equivalent valid GitHub Actions syntax).
4. Sonar step is **skipped** when the token is missing (conditional `if:`), so CI stays green before secrets are configured.

## verification

- `pytest` still passes locally / in CI with the new cov report flags.
- Workflow YAML validates (balanced keys, correct `uses:` pin).
- Grep: `ci.yml` contains `coverage.xml` and `sonarqube-scan-action`.
