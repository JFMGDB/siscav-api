## objective

Rate-limit **`POST /api/v1/login/refresh-token`** the same as login (**`5/minute`** per IP via **`slowapi`** / **`limiter`**, with **`Request` injected**), and **fail fast at startup** when **`ENVIRONMENT`** is **`production`** or **`prod`** (case-insensitive) and **`SECRET_KEY`** is missing, empty, or equals the literal **`change_me_in_development`**.

**Purpose:** Align refresh abuse surface with login (SEC-02) and block unsafe production signing configuration while preserving register → login → refresh flows (**AUTH-01**, **AUTH-02**).

**Output:** Decorated refresh handler, startup guard, and automated tests that prove limits and guard behavior without changing token TTLs or algorithm (**D-06**).

## must_haves

1. **`refresh_access_token`** declares **`@limiter.limit("5/minute")`** and includes **`request: Request`** as a parameter (same pattern as **`login_access_token`** in `auth.py`).
2. Sixth refresh request from the same test client identity/IP within one minute receives **429** (SlowAPI / **`RateLimitExceeded`** handling).
3. With **`ENVIRONMENT=production`** (or **`prod`**), importing/creating the app with invalid **`SECRET_KEY`** causes startup to fail with a **clear** error (process exit or exception before serving)—not a silent log only.
4. **`POST /api/v1/register`**, **`POST /api/v1/login/access-token`**, and **`POST /api/v1/login/refresh-token`** still complete successfully under normal test conditions (no regression of **AUTH-01**/**AUTH-02**).
